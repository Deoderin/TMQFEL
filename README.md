# TMQFEL

## Что делаем
Делаем MVP 2D-игры, вдохновленной King of Thieves. На первом этапе собираем локальный клиентский каркас без лишней сложности, но так, чтобы позже без перелома архитектуры подключить backend на endpoint API.

## Архитектурные принципы
- Один явный entrypoint приложения.
- Пока нет реальной необходимости, не вводим дополнительные слои и абстракции.
- Сцены и `MonoBehaviour` держим простыми и прозрачными.
- Усложняем архитектуру только когда появляется новая реальная ответственность.

## Текущее состояние
- Bootstrap-сцена: `InitialScene`.
- Основная игровая сцена: `GameScene`.
- `InitialPoint` должен жить в `InitialScene` и быть текущим runtime entrypoint.
- `InitialPoint` напрямую загружает `GameScene`.
- В проект добавлены `UniTask` и `DOTween`, ими можно пользоваться в задачах, где это реально упрощает код.
- Пока backend-код не вводим заранее. Добавим его, когда реально начнем подключать endpoints.

## Текущая архитектура
- `InitialPoint` (`MonoBehaviour`): единственный entrypoint, который на старте переводит игру из `InitialScene` в `GameScene`.

## Инструменты
- `UniTask`: для асинхронных игровых сценариев и последовательностей без корутин, когда это действительно упрощает код.
- `DOTween`: для анимаций, таймингов и простых scripted movement-последовательностей.
- Оба инструмента считаем доступными для дальнейших задач проекта.

## Правила для следующих шагов
- Новые сущности и слои добавляем только под конкретную задачу.
- Если появится реальная логика старта, тогда выделим отдельный startup flow.
- `README.md` поддерживаем как актуальный источник истины по игре, MVP-границам и текущей архитектуре.
