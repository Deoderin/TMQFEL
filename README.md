# TMQFEL

## Что делаем
Делаем MVP 2D-игры, вдохновленной King of Thieves. На первом этапе собираем локальный клиентский каркас без лишней сложности, но так, чтобы позже без перелома архитектуры подключить backend на endpoint API.

## Архитектурные принципы
- Один явный entrypoint приложения.
- Пока нет реальной необходимости, не вводим дополнительные слои и абстракции.
- Сцены и `MonoBehaviour` держим простыми и прозрачными.
- Усложняем архитектуру только когда появляется новая реальная ответственность.

## Текущее состояние
- Bootstrap-сцена: `InitialScene`.
- Основная игровая сцена: `GameScene`.
- `InitialPoint` должен жить в `InitialScene` и быть текущим runtime entrypoint.
- `InitialPoint` напрямую загружает `GameScene`.
- В проект добавлены `UniTask` и `DOTween`, ими можно пользоваться в задачах, где это реально упрощает код.
- Данные комнаты теперь храним в `LevelMatrixAsset` как матрицу уровня.
- Пока backend-код не вводим заранее. Добавим его, когда реально начнем подключать endpoints.

## Текущая архитектура
- `InitialPoint` (`MonoBehaviour`): единственный entrypoint, который на старте переводит игру из `InitialScene` в `GameScene`.
- `SystemsService` (singleton): простой runtime-реестр систем, через который можно получить нужный сервис по типу.
- `LevelMatrixAsset` (`ScriptableObject`): данные комнаты в виде редактируемой матрицы.
- `LevelMapConfig` (`ScriptableObject`): runtime-конфиг карты, который создается из матрицы и используется игрой. В нем же лежат ссылки на визуал уровня.
- `LevelCreator` (`MonoBehaviour`): собирает уровень в `GameScene` из `LevelMapConfig`.
- `LevelSystem` (plain class): прослойка над `LevelCreator`, через которую runtime-код получает доступ к данным и координатам уровня.
- `Player` (`MonoBehaviour`): сам игровой объект игрока с сериализованным визуалом и логикой спавна.
- `PlayerController` (`MonoBehaviour`): отдельный контроллер, который управляет `Player`.

## Инструменты
- `UniTask`: для асинхронных игровых сценариев и последовательностей без корутин, когда это действительно упрощает код.
- `DOTween`: для анимаций, таймингов и простых scripted movement-последовательностей.
- Оба инструмента считаем доступными для дальнейших задач проекта.

## Формат уровня
- Базовый размер матрицы уровня: `8 x 5`.
- Создание ассета: `Create/TMQFEL/Levels/Level Matrix`.
- Редактирование идет через custom inspector.
- Палитра выбирается цифрами `0-6` или кнопками в инспекторе.
- Левая кнопка мыши красит выбранным типом, правая очищает клетку в `0`.
- Кнопка `Save Config` создает или обновляет runtime-ассет рядом с матрицей.
- Типы клеток:
- `0`: `Empty`
- `1`: `Wall`
- `2`: `Monster`
- `3`: `Spawn`
- `4`: `Treasure`
- `5`: `Slope45UpRight` (`/`)
- `6`: `Slope45UpLeft` (`\`)

## Наклонные блоки
- Поддерживаем два типа наклонных блоков под `45°`.
- `Slope45UpRight` означает диагональ снизу-слева вверх-вправо.
- `Slope45UpLeft` означает диагональ снизу-справа вверх-влево.
- Эти блоки уже участвуют в runtime-вырезе переднего слоя и получают `PolygonCollider2D`.

## Runtime Config
- Имя runtime-конфига: `<LevelMatrixName>_Config.asset`.
- Конфиг хранит `width`, `height`, массив клеток и ссылки на `backgroundSprite`/`frontSprite`.
- Источник истины для редактирования уровня: `LevelMatrixAsset`.
- Источник данных для игры: `LevelMapConfig`.
- `Save Config` обновляет геометрию карты, но не должен мешать ручной настройке спрайтов в конфиге.

## Level Creator
- `LevelCreator` берет `LevelMapConfig` и строит уровень прямо в `GameScene`.
- `LevelCreator` не хранит ссылки на фоновые спрайты у себя, а берет их из `LevelMapConfig`.
- В `Awake` `LevelCreator` создает и регистрирует `LevelSystem` в `SystemsService`.
- Уровень имеет фиксированный world-size через `cellSize`.
- `LevelCreator` подгоняет только `orthographicSize` камеры, чтобы карта целиком влезала в кадр.
- Для fit используются настраиваемые отступы: `left`, `right`, `top`, `bottom`.
- За счет этого у всех игроков карта остается одного и того же размера в world coordinates.
- Сейчас генерация делится на 4 runtime-слоя:
- `BackgroundLayer`: задний слой комнаты.
- `CollidersLayer`: только коллайдеры по карте стен.
- `MarkersLayer`: временные маркеры для `Spawn`, `Monster`, `Treasure`.
- `FrontLayer`: одна цельная передняя картинка с alpha-вырезами по карте.
- `BackgroundLayer` и `FrontLayer` мыслятся как две большие картинки комнаты.
- Из `FrontLayer` вырезаются все пустые клетки, и через эти вырезы становится виден задний слой.
- Для slope-блоков вырез в переднем слое тоже идет по треугольной форме.
- соседние `Wall`-клетки в одной строке собираются в общие `BoxCollider2D`, чтобы убрать швы между тайловыми коллайдерами.
- `Slope45UpRight` и `Slope45UpLeft` получают `PolygonCollider2D`.
- Если назначен `frontSprite`, он режется по карте в runtime.
- Для `frontSprite` нужна включенная опция `Read/Write`, потому что генератор читает пиксели текстуры.
- Если пропорции экрана и карты не совпадают, меняется только камера, а не геометрия уровня.
- В `GameScene` `LevelCreator` уже повешен на объект `Envirement` и смотрит на `LevelMatrixGenerator_Config.asset`.
- `LevelSystem` отдает наружу `LevelMapConfig`, `cellSize`, мировую позицию клетки и стартовую позицию `Spawn`, чтобы runtime-объекты не работали с `LevelCreator` напрямую.

## Player
- Игрок сейчас снова добавлен в `GameScene` как отдельный root-объект `Player`.
- `Player` хранит сериализованные ссылки на визуал игрока и получает `LevelSystem` через `SystemsService`.
- `Player` также держит физическое тело `Rigidbody2D` и `BoxCollider2D`.
- `PlayerController` отдельно управляет движением игрока.
- Текущее поведение пока базовое:
- `Player` получает готовую стартовую позицию `Spawn` через `LevelSystem`;
- ставит игрока в стартовую позицию;
- `PlayerController` двигает его вперед по `moveDirection`;
- прыжок идет по `Space`, ЛКМ или тапу;
- рывок идет по `Left Shift` или ПКМ.
- если впереди стена, контроллер перестает задавать скорость в эту сторону;
- если игрок в воздухе упирается в стену, подъем вверх после прыжка не режется, а медленное скольжение начинается только на падении;
- прыжок теперь считается только от реального контакта с полом, а не от раннего down-cast, чтобы высота у стены была повторяемой;
- если во время рывка впереди препятствие, рывок останавливается.
- Сейчас это простой физический MVP без сложной логики прилипания к поверхностям.

## Правила для следующих шагов
- Новые сущности и слои добавляем только под конкретную задачу.
- Если появится реальная логика старта, тогда выделим отдельный startup flow.
- `README.md` поддерживаем как актуальный источник истины по игре, MVP-границам и текущей архитектуре.
